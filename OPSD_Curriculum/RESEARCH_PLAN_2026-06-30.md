> 상태 기록 + H100 신규 서버 작업 계획. 이 파일은 repo에 두어 H100 서버에서 `git pull`로 공유한다.

## 0. 한 줄 현황
원래 main이던 **level×subject 표현기하 커리큘럼(ours)** 은 난이도 baseline(diff)은 이기지만 **무작위-subject 대조(cond5)는 못 이김(seed0 기준)** → subject 순서 효과는 null에 가깝다. 대신 분석에서 **모델-불변의 "난이도 절벽" 구조**를 발견했고, 이를 살린 새 커리큘럼과 4B 실험으로 방향을 넓힌다.

## 1. 핵심 발견 (지금까지)

### 1-1. subject 축 (원래 가설) — 거의 null
- subjslack ours(α=2.0) full step900, avg@12:
  | bench | ours | diff | cond5(무작위 subject) |
  |---|---|---|---|
  | aime24 | 55.8 | 50.8 | 56.1 |
  | aime25 | 40.6 | 41.7 | 47.5 |
  | hmmt25 | 25.6 | 23.3 | 29.2 |
  | math500 | 85.0 | 79.6 | 87.0 |
- **ours > diff** (math500 +5.4 등) 이지만 **ours ≱ cond5**. → diff를 이긴 건 *난이도 스케줄 모양(overlap)* 덕이지 subject 기하 덕이 아님.
- **cond5 seed1/2 학습+eval 진행 중** (seed0 한 개로 확정 금지). 결과로 최종 판정.

### 1-2. 활성으로 난이도 재정의 — "오프라인 판정 불가"로 정정
- raw 활성은 난이도를 강하게 인코딩(level ρ≈0.71~0.84). **이건 진짜 신호다.**
- ⚠️ **중요 정정**: 활성-난이도가 response length와 얽힌 것은 *길이가 난이도의 증상*이기 때문(어려우면 답이 길어짐). 길이는 confound가 아니라 mediator → **회귀로 길이를 빼면 진짜 난이도 신호까지 버려진다.** 따라서 "길이 빼니 죽음 → 활성난이도 허상"은 **잘못된 검정**이었다.
- 다만 활성-난이도 ≈ 사람 level(0.84)이라 커리큘럼 축으로 쓰면 *대체로 level과 동일*. **level을 이기는지는 학습으로만 확인 가능**(오프라인 사망 선고 철회).

### 1-3. competence(모델 역량) — level과 갈리는 유일한 신호
- is_correct ~ level ρ = **−0.27** (8B·4B 동일). 모델 역량은 사람 라벨과 상당히 다름.
- pass-rate(2666) 분석: level 안 pass 산포 큼, **쉬운 라벨인데 모델 실패 18%, 어려운 라벨인데 성공 13%**, 역량순 정렬 시 **~52% 문제가 다른 stage로 이동**.
- 길이 제거 후에도 활성 주축이 competence를 인코딩(ρ 0.34~0.42). → competence는 robust한 별도 축.

### 1-4. 난이도 절벽 구조 — **가장 견고한 자산 (모델 불변)**
레벨 사이 활성 유사도 → 난이도가 두 덩어리 + 다리로 구성:
- **쉬운 클러스터 L1-2-3** (서로 유사) / **어려운 클러스터 L5-6-7-8** (서로 유사)
- **L4 = 외로운 다리(절벽)**. 인접 전이 거리: L3→L4, L4→L5가 압도적.
- **L3-L4-L5 구간이 전체 표현거리의 62~66%** 를 차지.
- **8B(62-65%)·4B(66%) 모두 동일**, pilot1/2 재현 r=0.996. → **모델·데이터 불변의 구조.**

## 2. 커리큘럼 설계

### 2-1. 설계 원칙 (확정)
1. **난이도 backbone = 사람 level 라벨 유지** (활성이 못 이김, 무라벨 비지도는 난이도 못 뽑음).
2. **클러스터를 쪼개지 마라** — L1-3, L5-8은 각각 한 덩어리로. (옛 ours 실패는 자연 구조를 가로질러 stage를 자른 것.)
3. **절벽(L3-L4-L5)에 커리큘럼을 집중** — 경계는 절벽에 맞추고, 거기서 더 오래/겹쳐서.
4. **"매끄러움 숫자"로 미리 판정하지 않는다** — 효과는 학습으로 확인.
5. **대조군 필수** — 새 커리큘럼은 항상 분포-매칭 무작위 대조(cond5식)와 같이 학습해 *구조 효과*를 분리.

### 2-2. 구조 살린 커리큘럼 "cliff" (main, **확정 2026-06-30**)
**7-stage 슬라이딩 창(너비3) + L8 보강 + dwell 균일.** 실험 모델 = **4B**(8B는 느려서 cond5 마무리만).

| stage | level | 비고 |
|---|---|---|
| S0 | L1-L3 | |
| S1 | L2-L4 | |
| S2 | L3-L5 | 절벽 한복판 |
| S3 | L4-L6 | |
| S4 | L5-L7 | |
| S5 | L6-L8 | |
| S6 | L7-L8 | L8 보강 |

- **난이도별 등장(구조)**: L1:1, L2:2, L3:3 L4:3 L5:3 L6:3 L7:3, L8:2.
- **dwell (gap-tilt + 하드 바닥, 최종)**: S0~S6 = 11/20/23/14/10/10/10%. 절벽(S1+S2) 43%, 하드끝(S4-S6) 31%.
  - **레벨별 유효 학습시간**: L1:4 L2:11 **L3:18 L4:19 L5:16** L6:12 L7:12 L8:9 (%). 절벽 정점 + 하드 보호.
  - dwell = 표현거리(gap)에 *약하게* 비례(균일 29%↔순수gap 63% 사이의 43%) → activation에 엮되 과반복(오버핏)·클러스터 과소학습 방지. 하드 클러스터엔 stage당 최소 ~10% 바닥.
- **overlap = 예/복습 (구조)**, **gap-tilt dwell = activation 노출 엮기 (시간)** — 둘은 별개 레버.
- **근거(activation)**: 레벨이 표현공간에서 두 클러스터(L1-3 / L5-8) + L4 절벽으로 갈림(§1-4). 강폭 넓은 절벽을 큰 점프 없이 건너려면 창을 거기서 더 겹쳐야(=촘촘한 디딤돌). "매끄러운 전이가 학습에 이롭다"는 가설 위에 성립 → 그 가설을 4B로 검증.
- 클러스터는 통째 유지(한가운데 경계 X). L8 보강은 끝점이 1× 되는 것 방지(역량은 L6-8에서 계속 하락하므로).

### 2-2b. 오염 정리 (2026-06-30, **완료**)
- eval(aime24/25·hmmt25·math500) ↔ 30K train 본문 대조: AIME/HMMT **0 겹침**. math500 진짜 중복 3개(+보수적 동일정답 고유사도 포함 **총 28개** 제거, 0.10%).
- **clean train = 29,406** (`scratchpad/clean_train_ids.csv`). 4B 학습은 clean셋 사용.
- 비교 영향: 오염은 모든 arm 공통 → arm 비교 무편향. 단 절대 math500은 미세 과대였음. 4B부터 제거.

### 2-3. 대조군 (cliff 효과 입증용)
- **cliff vs diff**(균일 level 밴드): 절벽 강조가 균일보다 나은가.
- **cliff vs 분포-매칭 무작위**: cliff와 같은 level 분포·dwell이되 stage 배정을 셔플 → *절벽 구조 자체*의 효과 분리.

### 2-4. 보조 변형 (여력 시)
- **activation-difficulty 커리큘럼**: 활성 난이도축으로 정렬(이제 사망 아님, 학습으로 level 대비 확인).
- **subject 재배치 변형**: subjslack과 다른 subject 배치(예: subject를 1차 축으로, 또는 subject-balanced) — cond5 결과 보고 가치 판단.
- **stage별 컨텍스트 확대**: 후반(어려운) stage에서 `max_completion_length`를 1024→2048/4096 램프. trainer는 `stage_index`가 이미 흐르므로 생성 길이를 stage별 lookup으로 바꾸는 소패치면 됨. (어려운 문제 reasoning 잘림 방지.)

## 4. H100 신규 서버 세팅 계획 (맨바닥)

조건: 아무것도 없음 / 인터넷·repo 접근 O / 지금은 **직접 실행**(SLURM 아님).

### 4-1. 세팅 순서
1. **conda 환경**: miniforge 설치 → `envs/opsd` 재현. (여기 서버의 env export로 `environment.yml`/`requirements.txt` 만들어 repo에 commit → H100에서 생성.)
2. **모델 가중치**: `Qwen/Qwen3-8B`, `Qwen/Qwen3-4B` (HF에서 다운로드, 인터넷 O라 가능).
3. **학습 데이터**: 28k universe(setA) + stage manifest(json). manifest는 repo에 있음(가벼움). 원본 row 테이블(`join_setA_rows.parquet`)·활성은 별도 전송 필요할 수 있음 → 크기 확인 후 결정.
4. **opsd_src**(vendored)는 repo에 있음 → pull로 확보.

### 4-2. GitHub 브릿지 워크플로 (클코 없이 협업)
1. (여기) 이식 가능한 학습 스크립트 + runbook commit.
2. (H100) `git pull` → 직접 실행.
3. (H100) eval json·로그만 push (큰 체크포인트 제외).
4. (여기) `git pull` → 분석/다음 설계.

### 4-3. 이식성 체크리스트 (스크립트 작성 시)
- 경로를 `$REPO`·`$HF_HOME` 등 환경변수로 파라미터화 (이 서버 절대경로 박지 말 것).
- 직접 실행용 `run_*.sh` (sbatch 헤더 없이, `accelerate launch ...` 직접) + SLURM 버전 별도.
- GPU 수(4×H100) 맞춰 `--num_processes`, batch/accum 조정.
- 데이터/가중치 존재 확인 가드(없으면 친절한 에러).
